1. CI/CD 实际会发生什么

NewAPI：main → 直接生产滚动

  • Go 文件变化会触发构建并调用源站 stack.sh ci-deploy newapi。
  • 生产 workflow 没有 go test、adapter contract test 或 candidate 流量验证。
  • scripts/** 不在触发 paths；SQL 和参数 seed 不会自动执行。
  • 容器 entrypoint 只启动程序；自定义商品 SQL 不随启动执行。

Adobe2API：main → candidate → promote

  • CI 先运行完整 pytest、shell syntax、compose config。
  • push main 默认部署 6002 candidate，不碰生产 6001。
  • candidate 会验证旧字段得到 422、multipart 严格契约和状态卷数量。
  • 只有手工 workflow_dispatch: promote 才切生产，失败可回滚。

4. 为什么 gpt-image-2-xk 会冲突

路由不是只看上游 model。 客户 public model 先翻译成唯一 internal model，再通过 abilities(group, model, channel_id) 选渠道，最后使用该渠道自己的 model_mapping 变成 Adobe2API 的 gpt-image。同一个 public 名若对应两个 enabled internal model，ResolveInternalModelName 会直接返回 ambiguous public model name，请求甚至到不了 abilities 路由；若强行复用同一个 internal model，则价格、UI profile 和商品语义无法按上游渠道分开。
现有 internal前缀剥离后的 public现状

cy-img2-gpt-image-2-2kgpt-image-2-2k已占用，且专门挂 Gulie 2K 清洗策略
cy-img2-gpt-image-2-4k / geek2-...gpt-image-2-4k已占用，现有 4K 商品/profile

5. 安全命名与路由方案

不要使用

adobe-gpt-image-2-1k

因为 adobe- 已作为渠道前缀被剥离，客户最终仍看到 gpt-image-2-1k，与现有商品碰撞。

cy-img2-gpt-image-2-xk

这是既有线路 B / Gulie / Geek2 的 internal 命名空间，不能被 Adobe profile 或校验接管。

推荐使用

internal: adobe-firefly-gpt-image-2-1k
public: firefly-gpt-image-2-1k
upstream: gpt-image

2K、4K 同理。public 名唯一，ModelPrice 独立，ability 只指向 Adobe channel 75。

adobe-firefly-nano-banana-pro-1k
→ firefly-nano-banana-pro-1k
→ nano-banana-pro

与 Adobe2API 文档中的 Firefly 能力语义一致，也不复用 Manju 商品价格。

6. 上线前必须完成的修正

  1. 把固定档位识别限定到 Adobe SKU。 不再用“包含 gpt-image/nano-banana + xK 后缀”的全局规则；只认 adobe-firefly-*（以及明确批准的 legacy Adobe alias）。
  2. 从 Adobe profiles 中删除 cy/go2api/geek2 match。 现有商品继续绑定原 profile；Adobe profile 只匹配 Adobe internal 名。
  3. 不要复用 Manju 商品做 Adobe 独立定价。 同一 internal model 只能有一个 ModelPrice；复用后无法按上游渠道分别售价。
  4. 为 NewAPI 增加发布门。 至少运行相关 Go tests、JSON 校验、Adobe adapter contract tests；最好先 candidate/影子实例。
  5. 数据库按 expand → deploy → enable 执行。 先写 profile/model/mapping/price（ability disabled），再部署兼容代码，验证后最后启用 abilities。
  6. 做真实请求矩阵。 每档覆盖 omitted、matching、wrong tier、prompt 写 4K、multipart edits、async submit/poll，并回归现有 gpt-image-2-xk。

7. 最终发布判断

Adobe2API 可以先合并吗？

可以合并到 main 生成 candidate；不要立即 promote。其严格接口已被 candidate contracts 覆盖，并且不会自动切生产 6001。

NewAPI 当前可以合并 main 吗？

不可以。 main 会直接滚动生产；全局 SKU 匹配和旧商品 profile 抢占必须先修正。

数据库 SQL 会随 CI 自动执行吗？

不会。当前 NewAPI workflow 只部署镜像，seed 与自定义迁移脚本需要独立、显式执行。因此必须把数据库变更纳入发布 runbook 或自动化 migration job。

证据来源：NewAPI production workflow、middleware public model translation、abilities 调度、model_mapping、imagevendor Gulie patch、UI profile seeder；Adobe2API candidate/promote workflow 与 strict contract checks。
